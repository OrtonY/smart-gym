# 饮食计划重构设计规格

日期：2026-06-07

## 背景

当前饮食模块已经支持食物图片或文字识别、手动记录、修正饮食日志和基础热量展示，但整体仍是“单条记录列表”。它不能像训练计划一样先生成计划、再根据用户反馈调整计划，也不能把当天实际饮食回写到计划状态中。用户能知道今天已经吃了多少千卡，但看不到“计划吃什么、实际吃了什么、偏差多少、近 7 天摄入趋势如何”的完整闭环。

本次重构将饮食模块调整为与训练计划类似的结构：AI 通过对话生成饮食计划，用户通过对话或手动编辑修改计划；热量识别和手动记录作为实际饮食事实；系统按用户本地时区在零点结算昨日饮食，把事实汇总回计划；前端展示今日饮食和近 7 天卡路里摄入量。

## 已确认方向

- 饮食计划按“天 + 餐次”组织。
- 默认生成 7 天饮食计划，但用户可以通过对话告诉 AI 生成几天。
- 用户可以在生成或调整对话中表达食物喜好、忌口、预算、口味、宏量营养目标等偏好。
- 每天默认包含早餐、午餐、晚餐和加餐。
- 热量识别时用户选择餐次，记录按日期和餐次自动归属到对应餐次计划。
- 零点同步的语义是结算昨日饮食，而不是每天自动改今天或未来计划。
- 今日和近 7 天卡路里摄入量是统计视图，固定按最近 7 个自然日展示，不受当前饮食计划天数限制。
- 生成上下文优先使用用户资料，生成对话中的天数和偏好覆盖或补充资料。
- 架构采用“饮食版训练计划模型”：计划是预期，`nutrition_logs` 是事实。

## 非目标

- 不在本轮建设完整食物库、食谱模板库、采购清单或库存管理。
- 不做真实硬件、第三方健康平台或外卖平台集成。
- 不自动修改今天和未来饮食计划；AI 调整必须由用户明确发起。
- 不删除或重写现有饮食日志；现有日志继续作为历史事实存在。
- 不把 `other` 餐次纳入默认计划生成；它仅用于用户额外记录。
- 不引入复杂营养医学建议或疾病饮食方案。

## 产品流程

用户进入饮食页后首先看到今日饮食。页面展示今日目标热量、实际摄入、宏量营养素、餐次完成状态和近 7 天卡路里摄入趋势。如果用户已有启用饮食计划，今日卡片来自计划中的当天餐次；如果没有计划，页面仍展示日志统计，并引导用户通过 AI 对话生成计划。

用户点击生成饮食计划后进入对话式输入。输入框默认提示生成 7 天计划，但用户可以直接说“生成 3 天”“未来 10 天低碳高蛋白”“早餐不要牛奶，午餐预算低一点”。后端把用户资料、最近饮食记录和这段对话合并为 AI 上下文，要求 AI 返回结构化 JSON。保存成功后创建一份饮食计划、版本 1 和对应餐次计划项。

用户需要修改计划时，通过同样的对话方式发起调整，例如“晚餐换成更清淡”“未来 3 天减少碳水”“不要西兰花”。后端读取当前计划和用户指令，创建新版本。旧版本保留，当前版本切换到新版本。

用户识别食物或手动记录饮食时，必须选择餐次和时间。后端根据 `logged_at` 的自然日、`meal_type` 和当前启用计划查找对应餐次计划，并把日志关联到该餐次。日志保存或修正后，后端立即重新计算该餐次和当天的实际摄入，让今日页面即时刷新。

每天 0 点系统按用户本地时区结算昨日饮食。结算服务汇总昨日所有饮食日志，更新昨日餐次计划的实际热量、宏量营养素和状态。没有日志的计划餐次标记为未记录；没有计划的日志仍作为独立事实参与近 7 天统计。

## 信息架构

### 饮食页

饮食页从“识别、手动、记录、心率”的功能入口集合，调整为今日饮食中心。

顶部展示今日汇总：

- 今日实际摄入 / 今日目标热量。
- 蛋白、碳水、脂肪实际值和目标值。
- 当前饮食计划名称、来源和覆盖日期。
- 识别食物、手动记录、生成或调整计划的主要操作。

中部展示今日餐次：

- 早餐、午餐、晚餐、加餐。
- 每餐的建议食物、份量说明、目标热量、宏量目标。
- 每餐实际摄入、偏差和状态。
- 单餐编辑、补记或查看关联日志入口。

趋势区域展示近 7 天卡路里摄入：

- 每个自然日的实际摄入。
- 如果当天有计划，展示目标热量作为对比。
- 明确区分“无记录”和“记录为 0”的情况。

记录区域保留已有饮食日志能力：

- 图片或文字识别。
- 手动记录。
- 日志列表。
- 识别结果修正。
- 修正后重新影响餐次和近 7 天统计。

### 资料页

用户资料继续保存身高、体重、健身目标、训练频率和饮食偏好。饮食计划生成默认读取这些字段。资料不足时，生成对话仍允许用户补充目标热量、食物喜好和限制条件。

## 数据模型

### 饮食计划

新增 `nutrition_plans`，保存用户的一份饮食计划：

- `id`
- `user_id`
- `title`
- `source`：`ai_generated`、`manual`
- `current_version`
- `is_active`
- `start_date`
- `end_date`
- `days_count`
- `created_at`
- `updated_at`

同一用户可以有多份计划，但同一时间只允许一份计划启用。创建新计划时，后端停用该用户其他启用计划，避免同一天同餐次存在多个启用计划。

### 饮食计划版本

新增 `nutrition_plan_versions`，保存计划版本：

- `id`
- `nutrition_plan_id`
- `version_number`
- `source`：`ai_generated`、`ai_adjusted`、`manual`
- `user_prompt`
- `change_summary`
- `created_at`

生成计划创建版本 1。AI 调整或批量手动替换餐次时创建新版本。旧版本不参与当前页面展示，但可以用于回溯。

### 餐次计划项

新增 `nutrition_plan_meals`，每条代表某天某餐：

- `id`
- `nutrition_plan_id`
- `version_number`
- `scheduled_date`
- `meal_type`：`breakfast`、`lunch`、`dinner`、`snack`
- `sort_order`
- `title`
- `food_items`
- `portion_notes`
- `target_calories_kcal`
- `target_protein_g`
- `target_carbs_g`
- `target_fat_g`
- `notes`
- `status`：`planned`、`logged`、`partial`、`over_target`、`missed`
- `actual_calories_kcal`
- `actual_protein_g`
- `actual_carbs_g`
- `actual_fat_g`
- `last_reconciled_at`

`food_items` 可以先用 JSON 数组保存结构化建议，例如食物名、份量和备注。第一版不要求关联食物库。

### 饮食日志扩展

现有 `nutrition_logs` 保留，并新增：

- `nutrition_plan_meal_id`

该字段可为空。日志创建时，如果存在当前启用计划中同日期、同餐次的餐次计划项，则自动关联；否则保持为空。历史日志不需要迁移补关联，但统计接口会按日期聚合所有日志。

## AI 生成与调整

新增饮食计划系统提示，要求 AI 只返回 JSON。返回内容包含标题、开始日期、天数、餐次列表和变更摘要。每个餐次必须包含日期、餐次、建议食物、份量说明、目标热量和宏量营养素。

生成默认天数是 7。用户 prompt 明确指定天数时，后端允许覆盖，第一版限制为 1 到 14 天。超过上限时返回 422，避免一次生成过大计划。

AI 上下文包括：

- 用户资料中的身高、体重、健身目标、训练频率、饮食偏好。
- 用户本次对话内容。
- 最近饮食记录摘要；如果没有历史记录，则传入空摘要。
- 当前日期和用户时区。

AI 调整读取当前计划版本和用户对话，返回完整的新版本餐次，而不是只返回差异补丁。这样后端保存逻辑与训练计划的 `replace items` 模式一致，降低局部 patch 出错风险。

## API 设计

### AI 饮食计划

`POST /api/ai/nutrition-plans/generate`

生成并保存一份饮食计划。请求包含 `prompt`、可选 `start_date`。如果 prompt 不指定天数，默认 7 天。后端从当前用户派生身份，不接受客户端传入 `user_id`。

`POST /api/ai/nutrition-plans/{plan_id}/adjust`

基于当前计划版本和用户 prompt 生成新版本。只允许调整当前用户自己的计划。

### 饮食计划

`GET /api/nutrition/plans`

返回当前用户饮食计划列表，按更新时间倒序。

`GET /api/nutrition/plans/{plan_id}`

返回计划详情、当前版本餐次和版本摘要。

`GET /api/nutrition/plans/{plan_id}/versions`

返回计划版本列表。

`PUT /api/nutrition/plans/{plan_id}/meals`

手动替换当前计划的餐次列表，创建新版本。该接口对齐训练计划的替换模式，适合批量编辑。

`PUT /api/nutrition/plans/{plan_id}/meals/{meal_id}`

只编辑单餐并创建新版本。今日餐次卡片的编辑入口使用该接口，避免为了改一餐要求前端提交整份计划。

### 饮食日志

现有接口继续保留：

- `POST /api/nutrition/logs`
- `POST /api/nutrition/recognize`
- `GET /api/nutrition/logs`
- `GET /api/nutrition/logs/{log_id}`
- `PUT /api/nutrition/logs/{log_id}/correction`

创建和修正日志后，后端重新计算关联餐次和当天统计。识别接口继续要求图片或文字至少一个存在。

### 统计与结算

`GET /api/nutrition/summary?days=7`

返回：

- 今日目标和实际摄入。
- 今日餐次计划及状态。
- 近 7 天目标与实际热量。
- 近 7 天宏量营养素汇总。

`POST /api/nutrition/reconcile`

执行当前用户饮食结算，用于测试、本地开发和用户侧补偿。生产环境每日 0 点任务调用同一个服务批量结算到点用户。

## 同步规则

所有自然日计算按用户资料里的 `timezone`。资料没有或时区无效时默认 `Asia/Shanghai`。

日志归属规则：

1. 把 `logged_at` 转成用户本地日期。
2. 查找当前启用计划当前版本中同 `scheduled_date`、同 `meal_type` 的餐次。
3. 如果存在，写入 `nutrition_plan_meal_id`。
4. 如果不存在，日志保持独立。
5. 创建或修正日志后重新计算该餐次和当天汇总。

餐次状态规则：

- `planned`：计划存在，尚未记录。
- `logged`：有记录，实际热量接近目标。
- `partial`：有记录，但明显低于目标。
- `over_target`：有记录，明显高于目标。
- `missed`：零点结算后，昨日计划餐次没有记录。

第一版阈值保持简单：实际热量低于目标 80% 为 `partial`，高于目标 120% 为 `over_target`，介于中间为 `logged`。目标为空时，有记录即 `logged`。

零点结算规则：

- 只结算昨日和更早未结算的餐次。
- 不自动修改今天和未来计划。
- 不删除历史日志。
- 没有计划的日志仍参与统计。
- 停用计划后，历史餐次和日志关联保持不变。

## 错误处理

- AI Provider 未配置时，生成、调整和识别返回明确错误；手动记录、读取计划和统计仍可用。
- AI 返回 JSON 不合法时不创建计划或新版本。
- AI 返回餐次为空、日期非法、餐次类型非法或目标热量非法时返回错误。
- 计划天数限制为 1 到 14 天。
- 客户端不能传入或覆盖 `user_id`。
- 读取、调整、编辑、结算都必须校验计划和日志属于当前用户。
- 日志修正导致日期或餐次变化时，需要重新归属并重新计算原餐次和新餐次。

## 前端设计

饮食页继续使用现有用户端布局和 Tailwind 风格，保持工作型界面，不做营销页。

页面主要区域：

- 今日汇总卡：实际/目标热量、宏量营养素、计划名称。
- 近 7 天图表：目标 vs 实际卡路里摄入。
- 今日餐次列表：早餐、午餐、晚餐、加餐。
- 对话式计划生成/调整面板。
- 食物识别和手动记录入口。
- 饮食记录与修正列表。

生成和调整输入不做复杂表单，使用一个对话输入框和少量默认提示。默认文案体现“默认生成 7 天，可直接说明天数和食物偏好”。

近 7 天图表使用轻量自绘条形图实现，不新增图表依赖。

## 测试计划

### 后端

- 模型测试：`nutrition_plans`、`nutrition_plan_versions`、`nutrition_plan_meals` 和 `nutrition_logs.nutrition_plan_meal_id`。
- AI 生成测试：fake AI 默认生成 7 天；prompt 指定 3 天时保存 3 天。
- AI 调整测试：调整计划创建新版本，旧版本保留，当前版本切换。
- 权限测试：用户不能读取、调整或结算他人的饮食计划。
- 日志归属测试：识别和手动记录按日期和餐次自动关联计划餐次。
- 日志修正测试：修正热量后餐次实际值更新；修正日期或餐次后重新归属。
- 结算测试：覆盖 `logged`、`partial`、`over_target`、`missed` 和无计划日志。
- 统计接口测试：返回今日和近 7 天目标/实际摄入。

### 前端

- API client 类型和构建检查。
- 饮食页加载 summary、计划和日志的空状态。
- 生成计划、调整计划、识别食物、手动记录和修正记录的主要交互。
- 近 7 天图表在无数据、部分数据和满 7 天数据下布局不崩。

### 回归

- `cd backend && pytest`
- `cd frontend && npm test`
- `cd frontend && npm run build`

## 实施顺序建议

1. 后端模型、迁移和 schemas。
2. 饮食计划 service 与基础 CRUD。
3. AI 生成和调整接口。
4. 日志归属、餐次聚合和结算服务。
5. summary 接口。
6. 前端饮食页重构。
7. 回归测试和视觉检查。
