# 训练闭环体验重构设计规格

日期：2026-06-07

## 背景

当前 Smart Gym 第一版已经具备基础可用能力，包括用户账号、训练课表、训练记录、动作库、姿态检测、AI 教练、饮食、榜单和轻量管理端。但用户端体验仍偏“功能入口集合”：首页展示多个入口，训练页偏记录表单，课表页承担了过多开始训练的职责，用户打开应用后不容易立刻知道今天该做什么、怎么开始、练完后下一步是什么。

本次重构以使用者视角优化训练闭环，参考 Keep 的核心体验：打开后看到今天可执行的训练，点击开始进入跟练，训练结束获得记录和复盘。范围控制为中等重构：新增训练模板数据源、今日训练聚合接口和跟练播放器，保留现有课表、动作库、姿态检测和训练记录的大部分模型。

## 已确认方向

- 主线：训练闭环优先。
- 首页：今日计划驱动，而不是功能入口列表。
- 今日训练来源：混合模式，优先读取用户当天课表；没有安排时推荐训练模板。
- 新增内容源：训练模板，而不是完整训练营体系。
- 开始训练形态：跟练播放器。
- 姿态检测：由用户在每次开始训练前选择是否开启；开启后本次训练内所有步骤都可叠加检测，不对每个动作重复征求同意。
- 课表与模板关系：模板只作为推荐和生成来源。模板加入课表时复制为课表动作项，课表不运行时引用模板，因为课表项必须允许用户自由修改。
- 训练计划与训练记录关系：课表是用户训练事实的主视图。系统每天 0 点按用户本地时区根据实际训练情况结算和更新课表，让计划日历同时体现已完成、部分完成、跳过、顺延、临时训练和未来安排。
- 实施范围：中等重构，新增必要数据模型和页面，不做全局产品大重写。

## 非目标

- 不在本轮引入训练营、会员课程、直播课、社交动态或复杂内容分发体系。
- 不把 Keep 的全部信息架构照搬到当前产品。
- 不重写 AI Provider、营养、榜单等非训练闭环模块。
- 不要求每个动作都有姿态检测规则；没有检测结果也不影响训练完成。
- 不强制用户把模板推荐加入课表后才能开始训练。

## 产品流程

用户进入首页后首先看到“今日训练”。系统先查用户当天课表，如果存在课表项，则展示当天训练；如果没有当天安排，则根据用户目标、训练偏好、历史训练和已发布模板推荐一个可立即开始的训练。

用户点击开始后进入训练概览页。概览页展示训练名称、预计时长、动作步骤、强度、器械或空间要求，并提供“开启姿态检测”的开关。用户选择开启时，前端请求摄像头授权；用户选择关闭或授权失败时，训练仍可按普通跟练继续。

跟练播放器按步骤推进，展示当前动作的视频或说明、动作提示、组数、次数、时长、休息和整体进度。本次训练开启姿态检测时，播放器叠加摄像头检测、计数和评分；未开启时，只按步骤完成训练。

训练结束后进入复盘页。复盘展示实际时长、完成率、消耗、步骤完成情况、姿态评分摘要和 AI 建议，并提供“安排下次训练”或“调整计划”的后续动作。训练完成会立即写入训练会话和步骤结果，并关联当天课表项；课表日历随后能直接展示当天训练事实。

每天 0 点系统按用户本地时区执行训练计划结算任务。它根据上一日课表项和实际训练会话更新课表状态：完成的保持完成，只有部分步骤完成的标记部分完成，完全未训练的按用户策略标记跳过或顺延，临时模板训练可以作为当天实际训练追加到计划日历中。这样课表不是静态待办，而是用户训练记录的日历化表达。

## 信息架构

### 首页

首页改为今日行动中心。主区域只突出今日训练，包括训练名称、来源、预计时长、动作数、强度、检测可用性和开始按钮。下方展示本周完成度、累计训练时长、平均评分和下一次训练推荐。

首页需要覆盖三种状态：

- 有当天课表：展示课表生成的今日训练。
- 无当天课表但有推荐模板：展示模板推荐训练。
- 无可推荐内容：提示暂无可用训练内容，并引导管理员发布模板或用户完善资料。

### 训练页

训练页从“新增训练表单”改为训练模板库、快速训练和历史记录。模板库支持按目标、难度、时长、目标部位筛选。用户可以直接开始模板训练，也可以把模板加入指定日期的课表。

### 课表页

课表页继续负责日历、AI 调整和手动编辑，同时也是训练记录的主视图。用户从模板加入课表后，课表保存展开后的动作步骤，用户可以修改动作、组数、次数、时长、休息和备注。模板后续变更不影响已生成的课表项。

课表日历需要展示每一天的计划状态和实际训练结果，包括未开始、已完成、部分完成、跳过、顺延和临时训练。当天训练结束后，课表项应能显示实际时长、完成率、评分摘要和关联复盘入口。每天 0 点结算后，昨日未完成项不应继续伪装成待办，而应按规则进入跳过或顺延状态。

### 跟练播放器

播放器是独立训练执行页面，消费后端返回的步骤快照。它支持上一项、暂停、继续、完成当前步骤、跳过当前步骤和结束训练。播放器内不再做模板推荐或课表编辑，保持执行流程聚焦。

### 练后复盘

复盘页展示本次训练结果，并把总训练记录、步骤结果和姿态检测结果聚合成用户可理解的摘要。AI 建议以训练改进和下次安排为主，不替代用户课表编辑。

## 数据模型

### 训练模板

新增 `workout_templates`，用于保存平台训练模板：

- `id`
- `slug`
- `title`
- `description`
- `goal`
- `difficulty`
- `target_muscles`
- `estimated_duration_minutes`
- `cover_url`
- `tags`
- `recommendation_weight`
- `is_published`
- `created_at`
- `updated_at`

模板对普通用户只暴露已发布内容；管理员可以维护全部模板。

### 模板步骤

新增 `workout_template_steps`，用于保存模板的动作步骤：

- `id`
- `workout_template_id`
- `sort_order`
- `exercise_id`
- `workout_mode_id`
- `title`
- `sets`
- `reps`
- `duration_seconds`
- `rest_seconds`
- `instruction`
- `allow_pose_detection`

步骤优先引用现有 `exercise_library`。没有对应动作时可以通过 `workout_mode_id` 表示宽泛训练模式。`allow_pose_detection` 只表示该步骤适合检测，是否开启由本次训练开始前的用户选择决定。

### 课表项扩展

现有 `training_plan_items` 保留，并扩展为能承载模板展开后的步骤信息：

- `duration_seconds`
- `rest_seconds`
- `instruction`
- `source_template_id`
- `source_template_step_id`
- `entry_type`：`scheduled`、`ad_hoc`
- `status`：`planned`、`completed`、`partial`、`skipped`、`rescheduled`
- `linked_workout_session_id`
- `completed_at`
- `actual_duration_seconds`
- `actual_score`

`source_template_id` 和 `source_template_step_id` 仅用于追溯来源，不作为运行时引用。用户修改课表项后，系统仍以课表项自身字段为准。

课表项承担计划与记录之间的桥接。训练会话保存详细事实，课表项保存日历视图需要的状态和摘要。每日结算任务只更新课表项状态和摘要，不改写历史训练会话。

`entry_type` 用于区分计划内安排和临时训练。用户提前加入课表或 AI 生成的课表项是 `scheduled`；用户当天直接开始模板或自由训练后追加到日历的记录项是 `ad_hoc`。临时训练项也使用 `completed` 或 `partial` 状态，但不会被当作未来待执行计划。

### 训练会话扩展

现有 `workout_sessions` 继续作为总训练记录，并扩展：

- `source_type`：`plan`、`template`、`free`
- `source_plan_id`
- `source_plan_item_id`
- `source_template_id`
- `pose_detection_enabled`
- `completed_steps_count`
- `total_steps_count`

当训练来源是课表时，`source_plan_item_id` 用于把本次训练会话关联回具体课表项。临时模板训练没有课表项来源，但可以在复盘或 0 点结算时作为 `entry_type=ad_hoc` 的当天实际训练追加到计划日历中。

### 用户时区

每日结算需要明确“用户本地 0 点”。现有 `user_profiles` 需要增加 `timezone` 字段，第一版默认 `Asia/Shanghai`。训练计划结算、今日训练聚合和训练记录归属日期都使用该时区计算自然日，避免跨时区时把训练归到错误日期。

### 计划结算状态

课表项状态用于让训练计划体现训练记录：

- `planned`：未来或当天尚未完成的计划。
- `completed`：关联训练会话达到完成标准。
- `partial`：有关联训练会话，但步骤完成率或时长不足。
- `skipped`：结算时没有实际训练，且不顺延。
- `rescheduled`：结算时未完成，并已顺延到后续日期。

第一版结算规则保持简单：有关联完成会话则完成；有关联但未完成全部步骤则部分完成；没有关联会话的昨日课表项按用户或系统默认策略标记跳过。顺延可以先作为显式用户操作或 AI 调整动作，不自动无限后移。

### 步骤结果

新增 `workout_session_steps`，保存每一步的完成结果：

- `id`
- `workout_session_id`
- `sort_order`
- `exercise_id`
- `workout_mode_id`
- `title`
- `planned_sets`
- `planned_reps`
- `planned_duration_seconds`
- `planned_rest_seconds`
- `actual_reps`
- `actual_duration_seconds`
- `score`
- `status`
- `pose_detection_result_id`
- `notes`

步骤结果保存的是训练开始时的快照，避免模板或课表后续修改影响历史记录。

## 今日训练聚合

新增 `TodayTrainingService`，负责生成首页和训练入口所需的统一响应。

聚合顺序：

1. 读取当前用户当天状态为 `planned` 或 `partial` 的有效课表项。
2. 如果有课表项，把课表项转换为 `TodayWorkout`。
3. 如果没有课表项，从已发布模板中按用户资料和历史训练推荐一个模板。
4. 如果没有可推荐模板，返回明确空状态。

已完成、已跳过或已顺延的课表项不再作为今日待训练内容返回，但可以作为首页训练摘要的一部分展示。

`TodayWorkout` 必须包含：

- `source_type`
- `source_id`
- `title`
- `description`
- `estimated_duration_minutes`
- `difficulty`
- `target_muscles`
- `steps`
- `pose_detection_available`
- `empty_state`

前端只消费 `TodayWorkout`，不在页面里重复实现“课表优先、模板兜底”的业务规则。

## API 设计

### 用户端

`GET /api/today/training`

返回当前用户今日训练。优先课表，兜底模板推荐，或返回空状态。

`GET /api/workout-templates`

返回已发布训练模板列表，支持目标、难度、时长、目标部位筛选。

`GET /api/workout-templates/{template_id}`

返回模板详情和步骤。

`POST /api/workout-templates/{template_id}/apply-to-plan`

把模板步骤复制到当前用户指定日期的课表。请求包含目标日期；后端从当前登录用户派生 `user_id`，不接受客户端传入用户身份。

`POST /api/workouts/sessions/start`

开始一次训练。请求包含来源类型、来源 ID 和本次是否开启姿态检测。后端创建训练会话并返回 `session_id` 与步骤快照。

`PUT /api/workouts/sessions/{session_id}/finish`

结束训练。请求提交实际时长、完成状态、消耗、步骤结果和姿态检测结果引用。后端保存总训练记录和步骤结果。如果训练来源是课表项，后端同步更新该课表项的完成状态和摘要。

`POST /api/training-plans/reconcile`

执行当前用户训练计划结算。该接口用于测试、本地开发和用户侧补偿。生产环境每日 0 点任务不依赖这个用户接口，而是调用同一个 `TrainingPlanReconciliationService` 批量结算到点用户。结算上一日计划，更新课表项状态，并把符合规则的临时训练追加为 `ad_hoc` 日历记录。

### 管理端

`GET /api/admin/workout-templates`

管理员读取全部模板。

`POST /api/admin/workout-templates`

创建模板和步骤。

`PUT /api/admin/workout-templates/{template_id}`

更新模板基础信息、步骤和发布状态。

## 姿态检测规则

用户每次点击开始训练前选择是否开启姿态检测。开启时，前端请求摄像头权限。授权成功后，本次训练内所有步骤都可以叠加检测，不再对每个动作重复确认。

后端不以“摄像头已授权”为权限来源，只记录本次训练是否启用姿态检测以及前端提交的实际检测结果。某个步骤没有检测结果时，步骤仍可标记完成。

姿态检测结果继续使用现有 `pose_detection_results`。步骤结果通过 `pose_detection_result_id` 关联检测结果，复盘页再聚合展示。

## 推荐策略

推荐策略必须读取课表结算状态。已完成或已部分完成的训练影响后续推荐，避免连续推荐同一肌群或强度；跳过和顺延状态也会影响推荐强度，避免用户因为漏练而被持续堆叠高强度训练。

第一版推荐策略保持简单、可解释：

- 优先匹配用户资料中的健身目标。
- 优先匹配用户训练频率和最近训练强度。
- 避免连续推荐相同目标肌群的高强度模板。
- 在同等条件下按 `recommendation_weight` 和最近未练模板排序。

如果用户资料不足，系统仍可以推荐通用入门模板，同时提示用户完善资料以获得更准确推荐。

## 管理端内容维护

管理端新增训练模板维护入口。管理员可以创建模板、调整步骤顺序、选择动作库动作、设置组数/次数/时长/休息、填写动作提示、设置难度和标签，并控制发布状态。

动作库仍负责单个动作的说明、媒体和检测规则。训练模板负责把动作组合成可跟练内容。

## 种子数据

本轮需要补充 6-8 个默认训练模板，覆盖：

- 入门下肢激活
- 核心稳定
- 上肢力量入门
- 全身燃脂
- 拉伸恢复
- 零器械快速训练

模板复用现有动作库，并为每个步骤提供清晰提示、预计时长、休息时间和检测可用性。

## 权限与数据隔离

- 普通用户只能读取已发布训练模板。
- 管理员可以维护全部模板。
- 课表、训练会话、步骤结果和姿态检测结果继续按 `user_id` 强隔离。
- 所有写入当前用户私有数据的接口都从 token 派生用户身份，不信任客户端传入 `user_id`。
- 模板复制到课表后成为用户私有课表项，后续模板编辑不会改写用户课表。

## 前端实现边界

前端新增或重构以下页面：

- 首页今日训练行动中心。
- 训练模板库和模板详情。
- 训练概览页。
- 跟练播放器页。
- 练后复盘页。

播放器页面只负责执行步骤、收集进度和调用姿态检测，不负责编辑课表。课表编辑仍留在课表页。

底部导航需要收敛用户端入口。首页、训练、课表、饮食、榜单、我的可以保留；AI Provider 设置不应作为普通用户底部主入口，建议移动到“我的”或设置页中。

## 测试策略

后端测试覆盖：

- 普通用户只能读取已发布模板。
- 管理员可以创建、更新、发布和下架模板。
- 模板复制到课表后保存为可编辑课表项。
- 修改模板不影响已生成课表项。
- 今日训练优先返回当天课表，没有课表时返回推荐模板。
- 今日训练不把已完成、跳过或顺延的课表项当作待训练内容。
- 无课表且无模板时返回明确空状态。
- 开始训练生成会话和步骤快照。
- 结束训练保存总记录、步骤结果和姿态检测结果引用，并同步更新关联课表项摘要。
- 每日 0 点结算任务按用户本地时区把昨日课表项更新为完成、部分完成、跳过或顺延。
- 临时模板训练可以作为 `ad_hoc` 训练记录显示在计划日历中。
- 用户资料时区默认 `Asia/Shanghai`，今日训练和每日结算按该时区归属训练日期。
- 用户不能访问或修改其他用户的课表、训练会话和步骤结果。

前端测试覆盖：

- 首页正确渲染今日课表、推荐模板和空状态。
- 模板库筛选和模板详情展示。
- 模板加入课表后进入可编辑课表项。
- 训练概览页的姿态检测开关和授权失败回退。
- 播放器步骤推进、暂停、跳过和结束训练。
- 练后复盘展示总结果和步骤结果。
- 课表日历展示计划状态、实际训练摘要和复盘入口。

## 交付边界

本设计完成后的实现计划应按阶段推进：

1. 后端模板模型、迁移、种子数据和管理接口。
2. 今日训练聚合接口和模板复制到课表。
3. 训练开始、步骤快照、结束训练、步骤结果和课表项状态同步。
4. 每日训练计划结算任务和课表日历记录视图。
5. 首页、模板库、训练概览、播放器和复盘前端。
6. 管理端模板维护和测试补齐。

每一阶段都应保持现有功能可用，避免一次性重写用户端全部模块。
